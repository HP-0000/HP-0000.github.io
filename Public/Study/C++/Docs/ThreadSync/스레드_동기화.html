동기화 알고리즘의 성능과 확장성

스레드를 사용할 때 성능에 가장 직접적인 영향은 캐시 라인의 변경이다. 읽기는 캐시 라인이 유지되며 병렬화 할 수록 성능은 증가한다. 쓰기의 경우 작업이 무거워 읽기 보다는 느리고, 만약 공유 자원에 대해서 쓴다면 캐시라인이 무효화 되기 때문에 성능이 확장되지 않는다. (그래프상 오히려 더 낮아지고 있음.) 하드웨어의 원자적연산(RMW)의 영향은 미미하다.

거짓 공유

관계 없는 두 데이터가 주소가 가까이 있다는 이유로 하나의 캐시라인으로 묶여 있어
멀티 스레드에서 캐시라인의 무효화가 발생하여 성능 하락을 유발함.

비트 쪼개기 전략

원자적 연산이 해결하는 공간은 하나지만 비트를 나눈 다면 여러 논리를 한 번에 처리할 수 있다.

CPU 코어 동작 이해

MESI 프로토콜

캐시 일관성을 위해 각 캐시라인이 갖는 상태

  • M (Modified): 나만 이 캐시를 가지고 있고, 메모리보다 최신 값임. (수정됨)
  • E (Exclusive): 나만 이 캐시를 가지고 있고, 메모리와 값이 같음. (독점)
  • S (Shared): 다른 코어도 이 캐시를 가지고 있고, 메모리와 값이 같음. (공유)
  • I (Invalid): 이 캐시 라인은 무효화됨. (읽으면 캐시 미스 발생)

각 코어는 캐시 라인을 업데이트 하기 위해, 지금 부터 너희 캐시라인은 무효화 하고 만약 최신 캐시를 가진 놈은 값을 내게 넘겨라고 하는 RFO(Read For Ownership)를 발송한다. 그 결과 다른 코어는 ACK를 응답을 보내고 캐시 라인의 상태를 I로 변경한다. 모든 ACK가 확인되고 최신 데이터가 넘어오면 해당 코어는 캐시 라인 상태를 E로 변경한다. 캐시라인을 업데이트 하고 M상태로 변경한다. 캐

캐시 라인 상태가 I 가 된 코어는, 최신 상태의 코어의 캐시라인을 가져오고, 각각 S 상태가 된다.

Invalid 처리하느라 코어가 바로 멈추는게 좀 그렇다.

코어는 ACK 응답을 바로 발송하되 바로 캐시 라인을 무효화 처리하지 않고 Invalid Queue 에 추가해 둔다.
여유가 되면 그제서야 캐시 미스를 발생시킨다. 만약 RFO 로 최신 상태 값을 가져오려 한다면, Invalid queue 속 관련 메시지는 무시하거나 먼저 처리한다.

ACK를 기다리느라 코어가 멈추는게 좀 그렇다

코어는 캐시에 바로 쓰는 대신에 ACK를 기다리지 않고 Store Buffer에 기록한다.
곧바로 해당 값을 읽어야 하는 상황이라면, Store buffer에서 읽어온다.
ACK가 다 오면 그제서야 Store Buffer 값은 캐시 라인으로 업데이트 된다.

이때 최적화가 일어나면서 Store Buffer는 비 순차적 구조로, 캐시 라인을 업데이트 한다.
프로그래밍 결과만 같게 보인다면야, 서로 다른 캐시라인들은 묶어서 처리하는게 좋기 때문이다.

Load 값을 Store 하는 중간에 Invalid queue 를 처리하면 과거 Load 값을 Store 하면 안되는데

Read-Modify-Write 를 원자적 절차로 해결 하기 위해 RFO를 잡고 나서 Load 와 Write를 진행하는 잠금 연산을 사용한다.

임계 구역 설정

RMW 로 락 변수를 1로 설정한 뒤 Invalid Queue를 비워 캐시라인을 최신화 한다. 이러면 임계구역 진입이다.
다른 변수는 락 변수가 0인 경우만 1이 될 수 있으며 아니라면 소프트웨어적으로 스핀 루프를 돌리던가 운영체제가 잠시 스레드를 재우던가 해서 임계구역에 진입할 때 까지 시도를 해야한다.

임계 구역에서 명령을 다 실행한 다음에는 Store Buffer를 캐시 라인에 반영한 뒤 락 변수를 0으로 만들어 임계 구역을 해제한다.

메모리 배리어

컴파일러와 cpu 최적화에 대해서, 다른 코어를 위해 명령어 순서와 가시성을 보장하기 위한 명령

LoadLoad (읽기-읽기 차단) 

앞의 읽기가 끝나기 전에, 뒤의 읽기가 실행되는 것 방지
읽기 후 Invalidate Queue를 정리하고 이후 읽기를 진행한다.

LoadStore (읽기-쓰기 차단) 

앞의 읽기가 끝나기 전에, 뒤의 쓰기가 캐시에 반영되는 것 방지
Store buffer 값을 캐시로 내보내기전 쓰기에 Dependency 가 걸려 Load 가 완료될 때까지는 Stall 된다.
Store 가 stall 걸리는건 아니다.

StoreStore (쓰기-쓰기 차단) 

앞의 쓰기가 캐시에 가기 전에, 뒤의 쓰기가 캐시에 반영되는 것 방지
Store buffer 의 캐시 반영 순서는 앞의 쓰기 이후 뒤의 쓰기가 된다.

StoreLoad (쓰기-읽기 차단) 

앞의 쓰기가 캐시에 가기 전에, 뒤의 읽기가 실행되는 것 방지
Store buffer가 캐시에 반영될 때까지 Load를 Stall 한다.
